      20             :  *  The dereferencing code's purpose is to take a symbol with pointer type that
      21             :  *  is being dereferenced, and translate it to an expression representing the
      22             :  *  values it read or wrote to/from. Note that dereferences can be
      23             :  *  'dereference2t' expressions, or indexes that have a base with pointer type.
      24             :  *
      25             :  *  The inputs to this process are an expression that contains a dereference
      26             :  *  that is to be translated, and a value_sett object that contains the set of
      27             :  *  references that pointers may point at.
      28             :  *
      29             :  *  The output is an expression referring that nondeterministically evaluates
      30             :  *  to the piece of data that the dereference points at. This (usually) takes
      31             :  *  the shape of a large chain of 'if' expressions, with appropriate guards to
      32             :  *  switch between which data object the pointer points at, according to the
      33             :  *  model that the SAT solver is building. If the pointer points at something
      34             :  *  unknown (or dereferencing fails), then it evalates to a 'failed symbol', a
      35             :  *  free variable that isn't used anywhere else.
      36             :  *
      37             :  *  The dereferencing process also outputs a series of dereference assertions,
      38             :  *  the failure of which indicate that the dereference performed is invalid in
      39             :  *  some way, and that the data used  (almost certainly) ends up being a free
      40             :  *  variablea free variable. These are recorded by calling methods in the
      41             :  *  dereference_callbackt object. Assertions can be disabled with the
      42             :  *  --no-pointer-check and --no-align-check options, but code modelled then
      43             :  *  relies on undefined behaviour.
      44             :  *
      45             :  *  There are four steps to the dereferencing process:
      46             :  *   1) Interpretation of the expression surrounding the dereference. The
      47             :  *      presence of tertiary operators, short-circuit logic and so forth can
      48             :  *      affect whether a dereference is performed, and thus whether assertions
      49             :  *      fire. Identifying index/member operations applied also helps to simplify
      50             :  *      the resulting dereference.

      54             :  *      pointer variable points at that data object, which are used to combine

      57             :  *      object, and a (possibly nondeterminstic) offset, this code must create
      58             :  *      an expression that evaluates to the value in the data object, at the
      59             :  *      given offset, with the desired type of this dereference.
      60             :  *   4) Assertions: We encode assertion that the offset being accessed lies
      61             :  *      withing the data object being referred to, but also that the access has
      62             :  *      the correct alignment, and doesn't access padding bytes.
      63             :  *
      64             :  *  A particular point of interest is the byte layout of the data objects we're
      65             :  *  dealing with. The underlying SMT objects mean nothing during byte
      66             :  *  addressing, only the code in 'type_byte_size' is relevant. To aid
      67             :  *  dereferencing, that code ensures that all data objects are word aligned in

      69             :  *  can avoid all scenarios where dereferences cross field boundries, as that
      70             :  *  will ultimately be an alignment violation.
      71             :  *
      72             :  *  The value set tracking code also maintains a 'minimum alignment' piece of
      73             :  *  data when the offset is nondeterministic, guarenteeing that the offset used

      75             :  *  number of behaviours we have to support when dereferencing, particularly
      76             :  *  useful when accessing array elements.
      77             :  *
      78             :  *  The majority of code is in the reference building stuff. It would be aimless
      79             :  *  to document all it's ecentricities, but here are a few guidelines:
      80             :  *   * There are two different flavours of offsets, the offset of the pointer
      81             :  *     being dereferenced, and the offset caused by the expression that's
      82             :  *     dereferencing it.
      83             :  *   * 'Scalar step lists' are a list of expressions applied to a struct or
      84             :  *     array to access a scalar within the base object. For example, the expr
      85             :  *     "foo->bar[baz]" dereferences foo and applies a member then index expr.
      86             :  *     The idea behind this is that we can directly pull the scalar value out
